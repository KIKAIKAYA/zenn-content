## AI に長い作業をさせると、毎回同じ 3 つの壁に当たる

実務で AI エージェントに多段の作業（調査・生成・レビュー・報告のような流れ）をさせると、
だいたい同じところで転びます。

1. **セッションが死ぬと全部やり直し。** 200 件の反復処理の 150 件目で切れたら、150 件を捨てるのか？
2. **「できました」が信用できない。** LLM の自己申告の合否は、合っているときと合っていないときがある。
3. **どこにいくらかかったか分からない。** 終わってみたらトークンが溶けている。

この 3 つは「プロンプトを工夫する」では解決しません。**実行基盤の設計問題**です。

私はレガシーマイグレーションの現場で AI ワークフローを運用してきた経験から、
この 3 つに対する設計回答として [flowsmith](https://github.com/KIKAIKAYA/flowsmith) という
Claude Code 向けのワークフローエンジンをゼロから設計・実装しました。
この記事はその設計判断の記録です。

## 解決の方向：制御は宣言に、知能はステップの中に

ReAct のように「LLM が次の一手を考えながら進む」方式は柔軟ですが、
**同じ依頼で二度走らせると違う経路を通る**という根本的な不安定さがあります。

flowsmith は逆に振りました。

```yaml
steps:
  - id: survey          # 作る（make）
  - id: review          # 作って検査、不合格なら差し戻し（make-check）
  - id: accept_gate     # 判定して分岐、ダメなら前工程へ（branch）
  - id: report
```

- **経路・合否・ルーティングは人間が YAML に宣言**し、エンジンが機械的に実行する
- **各ステップの中身**（コードを読む、表を書く、レビューする）だけを LLM の subagent がやる

「予測可能性 > 自由度」。エージェントの知能はステップの内側に閉じ込めて、
フロー制御には一切使わない、という割り切りです。

## 設計判断 1：合否を LLM に決めさせない

一番大事な判断です。レビュー担当（checker）にはこう返させます：

```
verdicts:
  - {id: V01, severity: high, verdict: pass}
  - {id: V02, severity: mid,  verdict: fail}
```

checker が返すのは**観点ごとの構造化された判定だけ**。
「合格です／不合格です」という総合結論は返させても**使いません**。
エンジンが `pass_when: no_high_fail` のようなルールを verdicts に機械適用して合否を決めます。

さらにその手前に**機械ゲート**を置きます。ファイルが存在するか、必須見出しがあるか、
表の行数が上流の件数と合っているか——こうした決定的に判定できる検査を先に走らせ、
落ちたら LLM レビューに進ませない。**機械で落とせるものを LLM に回さない。**
LLM レビューは高くてランダムですが、機械検査はタダで確実だからです。

## 設計判断 2：状態はすべてディスクに置く

セッションはいつ死んでもいい、という前提で設計します。

```text
${OUT}/.flowsmith/
├── state.json   # どのステップ・どの行まで終わったか
├── run.jsonl    # 流水ログ（所要時間・トークン込み）
└── runs/        # 完了フローのアーカイブ
```

state.json は**ステップ完了ごと・反復の 1 行完了ごと**に更新します。
再実行すると、エンジンは state を検出して完了分をスキップし、続きから走ります。

反復処理を行単位で記録するのがポイントで、150/200 件で死んでも残り 50 件だけ再開できます。

## 設計判断 3：行き詰まったら止まる（seized）

結果が解析できない、再試行上限を超えた、ルーティング先が不正——
こういうとき、エンジンは**推測で進まず停止**します。状態名は `seized`（焼き付き）。

停止時には「どの観点が・何回・どんな根拠で通らなかったか、人間の選択肢は何か」という
行き詰まりサマリーを出して人間に裁定を返します。

黙って間違った道を進むコストは、止まるコストの百倍つきます。
**人間と AI の接縫は、暗黙にしないで明示的に設計する**——これが全体を貫く方針です。

同じ理由で、レビュー・反復・戻りジャンプにはすべて上限があり、超過は必ず seized になります。
「いつまでも回り続ける」状態が構造的に存在できません。

## 実験で出た、いちばん面白いバグ：差し戻しの誤読

branch ステップには carry という仕組みがあります。判定者（arbiter）が「no」と言った理由を、
差し戻し先のステップに引き継ぐ機能です。

最初の実装では、理由文を「前回の判定理由を踏まえてください」という中立的な文面で渡していました。
すると何が起きたか——

**作業者（maker）が、不合格理由を「あるべき姿の説明」と誤読して、
「既に要件を満たしています。変更不要です」と無変更で返してきた**のです。

不合格理由には「ライセンスへの言及がない」と書いてある。maker はそれを読んで
「ライセンスに言及すべきなのだな、（自分の成果物は）大丈夫だな」と解釈した。
人間の新人でも起きそうな誤読です。

修正は文面の明示化でした：

```
## ⚠ 差し戻し（前回の成果物は判定で不合格になった）
上記の理由は「現状の説明」ではなく「解消すべき不足」である。
「変更不要」という結論は禁止。
```

ここまで書いて、初めて確実に直るようになりました。
**マルチエージェントの結合部では、情報の「内容」だけでなく「フレーミング」が動作を決める**——
これがこの実験から得た一番の教訓です。

## 実負荷での検証：レガシー調査 15 ステップを完走

設計が机上で終わらないことを確認するため、実ユースケースで検証しました。
[migration-factory](https://github.com/KIKAIKAYA/migration-factory) のレガシー現状調査フロー
（15 ステップ：資産棚卸し → 分類 → 依存抽出 → 影響分析 → 検収ゲート → 二層報告書）を、
JPetStore 6（160 ファイル / 15,007 行）に対して実行しました。

- **事前文脈なしの新規セッションで全 15 ステップ完走**（約 35 分）
- **途中 2 回中断 → 2 回とも続きから再開**
- **差し戻しが実際に発動**：影響分析と報告書の 2 ステップが round 1 不合格 →
  指摘付きで再実行 → round 2 合格
- 報告書の見出しが規定と 1 文字違っただけで機械ゲートが差し戻した場面もありました
  （厳しすぎる気もしますが、「機械は厳密」の側に倒しておくほうが安全です）

出力一式（CSV 17 本・報告書・全レビュー表・実行ログ）はそのまま
[公開しています](https://github.com/KIKAIKAYA/migration-factory/tree/main/examples/jpetstore)。
差し戻しの経緯もログとレビュー表で追えます。

そしてこの実負荷検証は、**エンジン自身の欠陥**もひとつ暴きました。
公開した実行ログを見ると、時刻が分単位の丸い値ばかりで、トークン数の欄もありません。
プロンプト実装のエンジン（エンジン自体が Claude）は、正確な計時・トークン集計という
「機械的な記帳」を確実には実行できないのです。——これは「機械で判定できるものを LLM にやらせない」
という本記事の主張がエンジン自身にも刺さることを意味します。決定的な部分（YAML 解析・状態管理・
ゲート判定・ログ記録）をコードに置き換える計画の、いちばん説得力のある根拠が、
自分のログの不正確さだったというわけです。

## まとめ：エンジンは「賢く」しない

flowsmith の設計を一言でいうと「**エンジンを意図的に愚直にする**」です。

- 経路は YAML の宣言どおりにしか動かない
- 合否は構造化データに機械的にルールを当てるだけ
- 迷ったら止まって人間に返す
- 知能が必要な仕事だけを、ステップの中の LLM に渡す

AI エージェントの信頼性は、エージェント自身の賢さではなく、
**賢さを使う場所を限定する設計**から来る——というのが、作ってみての実感です。
