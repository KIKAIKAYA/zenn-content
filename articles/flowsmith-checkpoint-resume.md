## 200 件の反復処理が 150 件目で死んだら、150 件を捨てるのか

AI エージェントに長い作業をさせると、いつか必ずセッションが死にます。
コンテキスト上限、レート制限、ネットワーク、人間都合の中断——理由は何でもいい。
問題は死ぬことではなく、**死んだあと全部やり直しになる**ことです。

[前回の記事](https://zenn.dev/kikaikaya/articles/flowsmith-declarative-ai-workflow)で書いた
ワークフローエンジン [flowsmith](https://github.com/kikaikaya/flowsmith) は、
「セッションはいつ死んでもいい」を設計の前提に置きました。
この記事はそのチェックポイント機構の設計判断——特に、設計中いちばん悩んだ
**「やりかけの半端な成果物」をどう扱うか**——の記録です。

## 状態はすべてディスクに置く

エンジンは実行状態を `state.json` という 1 ファイルに永続化します。

```json
{
  "flow_id": "legacy_survey",
  "status": "running",
  "completed_steps": ["asset_check", "asset_inventory"],
  "next_step": "layer_map",
  "steps": {
    "asset_inventory": { "status": "done", "outputs": ["data/01-inventory.csv"] }
  }
}
```

再実行時、エンジンはまず state.json を探します。あれば `completed_steps` をスキップして
`next_step` から続行。なければ新規実行。**ユーザーの操作は「同じコマンドをもう一度」だけ**です。

ポイントは更新タイミングです：

- **ステップが完了するたび**に更新する
- 反復処理（foreach）では**1 行完了するたび**に更新する

後者が効きます。200 件の反復の 150 件目で死んでも、state には 150 件分の完了キーが
残っているので、再開時は**残り 50 件だけ**を実行します。

## いちばん悩んだ問題：「半端な成果物」は使えるのか

ここが本題です。ステップの途中でセッションが死んだとき、
書きかけのファイルがディスクに残ることがあります。これをどう扱うか。

「途中まで終わっているなら、続きから書き足せば節約になるのでは」——最初はそう考えました。
しかし採用しなかった。理由は単純で、**半端な成果物が使える状態かどうか、誰にも分からない**からです。
表の途中で切れているのか、整合性が壊れているのか、検査する手段がない。
「たぶん大丈夫」で続きを書き足して壊れたデータが下流に流れるコストは、
作り直すコストよりはるかに高い。

そこで設計はこうなりました：

1. **ステップの状態は done か not-done の二値だけ。**「半分終わった」という状態は存在しない
2. done になる条件は厳格——エージェントが規定形式の結果ブロックを返し、
   **かつ**機械ゲート（ファイル存在・必須見出し・件数照合）を通過したときだけ
3. 途中で死んだステップは next_step に残ったまま。再開時は**ステップの頭から作り直し**
4. 作り直し時、エージェントは固定パスに**上書き**で出力する。半端なファイルは丸ごと消える

つまり「**疑わしい半端は、検査せずに捨てる**」。
不確実なものを賢く救済しようとせず、確実なものだけを信じる——
合否判定を LLM の自己申告に任せない（前回記事）のと同じ思想です。

この設計には守るべき不変条件がひとつあります：
**エージェントの出力は必ず上書き（overwrite）、追記（append）は禁止**。
どこかのステップが追記方式になった瞬間、「半端＋作り直し」が混ざって汚染が起きます。

## トレードオフも正直に書く

この設計の弱点は、**作り直しの粒度が「ステップ全体」**なことです。
大きなステップが 90% まで進んで死ぬと、90% 分のトークンを捨てて最初からやり直します。

緩和策が foreach の行単位チェックポイントです。反復系の重いステップは行ごとに
完了を記録するので、捨てるのは最大でも「行 1 つ分」。
実験では 6 件の反復を 3 件目で意図的に中断し、再開後に残り 3 件だけが実行されること、
1 件目のファイルのタイムスタンプが変わっていない（再実行されていない）ことを確認しました。

```jsonl
{"event":"flow_resume","flow":"rows","skipped":["write_notes: 3/6 行完了済み"]}
```

一方、反復構造を持たない大きなステップ（たとえばコード全体の依存関係抽出）は
行単位に割れません。ここは現状の設計の限界として残っています。

## 実負荷で 2 回死んで、2 回復帰した

この機構は実験用の小さなフローだけでなく、実負荷でも検証済みです。
レガシー調査フロー（15 ステップ）を JPetStore 6 に対して実行したとき、
レート制限などでセッションが**2 回**死にました。実行ログ（抜粋）：

```jsonl
{"event":"flow_resume","from_step":"asset_inventory","completed_steps":["asset_check"]}
{"event":"flow_resume","from_step":"flow_trace","completed_steps":[...11 ステップ分...]}
```

1 回目は序盤（1 ステップ完了時点）、2 回目は終盤（11 ステップ完了時点）。
どちらも「同じコマンドをもう一度」で完了済みステップをスキップし、続きから完走しました。
中断を挟んだ合計でも全 15 ステップ・約 35 分（Claude Code の実測作業時間）。
ログと成果物一式は[公開しています](https://github.com/kikaikaya/migration-factory/tree/main/examples/jpetstore)。

意図的なテストではなく、**本物のレート制限が本物の中断として襲ってきて、設計どおりに復帰した**——
チェックポイント機構の検証としては、これ以上ない実戦でした。

## まとめ：完了だけを信じる

flowsmith のチェックポイント設計は 3 行に要約できます。

- **状態はすべてディスクに**（セッションはいつ死んでもいい）
- **完了だけを信じる**（done の条件は結果ブロック + 機械ゲートの二段）
- **半端は検査せず捨てる**（上書きで作り直す。救済しない）

「賢く救済する」より「愚直に作り直す」ほうが、総コストは安い——
小さくない反復作業を AI に任せるなら、この割り切りが効きます。
